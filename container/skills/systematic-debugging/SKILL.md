# 结构化 Debug 工作流

强制按科学方法排查问题，禁止凭直觉瞎猜瞎改。每一步必须有明确输出，不跳步。

## 核心原则

- **一次只改一个变量**：每次验证只测一个假说，不要同时改两个东西
- **先观察后行动**：收集完所有症状再动手，不要看到第一个线索就冲上去改
- **排除法优于猜测**：用实验排除假说，不要"我觉得应该是这个"
- **记录每一步**：每个假说的验证结果都要记下来，避免重复验证

## 执行步骤

### Step 1: 收集症状

在动手之前，先回答以下问题（信息不足的主动问用户）：

1. **错误现象**：具体报什么错？完整的错误信息是什么？
2. **复现条件**：每次都能复现还是偶发？复现步骤是什么？
3. **环境信息**：OS、运行时版本、依赖版本、部署环境
4. **时间线**：什么时候开始出问题的？之前有改过什么？
5. **影响范围**：只影响一个地方还是多处？有没有正常工作的对照组？

**输出格式**：

```
## 症状摘要
- 错误信息：[精确引用]
- 复现率：[必现/偶发/特定条件]
- 首次出现：[时间点或触发事件]
- 最近变更：[相关的代码/配置/环境变更]
- 对照组：[哪些场景是正常的]
```

### Step 2: 形成假说

基于症状列出**至少 3 个**可能的根因，按概率从高到低排序。

对每个假说必须说明：
- **假说内容**：一句话描述可能的原因
- **支持证据**：哪些症状支持这个假说
- **反对证据**：哪些症状与这个假说矛盾
- **验证方法**：怎么用最小代价验证或排除

**输出格式**：

```
## 假说列表
### H1: [概率最高的假说] — 可能性 ★★★
- 支持：...
- 反对：...
- 验证：...

### H2: [次高] — 可能性 ★★
- 支持：...
- 反对：...
- 验证：...

### H3: [第三] — 可能性 ★
- 支持：...
- 反对：...
- 验证：...
```

### Step 3: 逐个验证

从概率最高的假说开始，逐个验证。

**每个验证必须遵守**：
1. **只改一个变量** — 一次实验只测一个假说
2. **设计最小实验** — 用最少的改动验证，不要顺手重构
3. **记录结果** — 不管验证通过还是排除，都要记录

**输出格式（每个假说验证后更新）**：

```
### H1 验证
- 实验：[做了什么]
- 结果：[观察到什么]
- 结论：✅ 确认 / ❌ 排除 / ⚠️ 部分相关
```

如果前 3 个假说都被排除，回到 Step 2 补充新假说。

### Step 4: 定位根因

综合所有验证结果，明确指出根因。

**必须包含**：
- **根因**：一句话描述
- **证据链**：哪些实验结果指向这个结论
- **排除项**：哪些假说被排除了，为什么

**禁止**：
- 不要说"可能是 A 也可能是 B"——必须给出确定结论
- 如果真的无法确定，说明还需要什么信息才能确定

### Step 5: 修复 + 回归

1. **最小修复**：只改导致问题的部分，不要顺手改别的
2. **验证修复**：用 Step 1 的复现步骤确认问题已解决
3. **回归测试**：检查修复是否引入新问题
   - 相关功能是否正常
   - 已有测试是否通过
   - 边界条件是否覆盖

**输出格式**：

```
## 修复报告
- 根因：[一句话]
- 修复内容：[改了什么，为什么这么改]
- 验证结果：[原问题是否解决]
- 回归检查：[有无引入新问题]
```

## 注意事项

- 如果用户说"帮我看看这个 bug"，不要直接开始改代码，先走 Step 1 收集症状
- 如果用户已经给了充分的错误信息，可以快速过 Step 1，但不能跳过
- 追踪调用链时必须追到实际 I/O 边界（HTTP 请求、文件读写、子进程 spawn），不要在中间层停下来
- 每个 Step 完成后向用户汇报进展，不要闷头干到最后才说
